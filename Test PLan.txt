Restful-Booker Test Plan
Created by Тупицын Максим Васильевич

Вступление:
Данный артефакт является финальным заданием курса 'Функциональное тестирование ПО' от образовательного центра 'IT-Academy'

Предмет тестирования:

Restful-Booker сервис бронирования гостиниц, который позволяет выполнять следующие действия:
-Создать бронирование
-Считать бронирование
-Обновить бронирование (все поля)(требуется авторизация)
-Обновить бронирование (частично)(требуется авторизация)
-Удалить бронирование(требуется авторизация)
-Доступность сервиса
-Авторизация в сервисе для выполнения обновления/удаления

В ходе тестирования будет проверен API,включая: 
-Validation testing
-Functional testing(частично)
-Load testing(дополнительно)
-Perfomance(частичная проверка)

В тестирование не включаются следующие проверки:
-UI testing
-Security testing
-Penetration testing
-Fuzz testing
-Runtime and error detection

Тестирование будет проводится на:
-Windows 11/64(last public build)

Инструменты используемые для тестирования:
-Postman(коллекция запросов)
-TestRail(Тест сьюит + коллекция тест кейсов + тест ран)
-Jira(Отчетность+ссылка на GitHub)
-Jmeter(нагрузочное тестирование)

Стратегия тестирования
--Входные критерии:--
-Сервис доступен
-Сервис не получает новых коммитов в течении тестирования
-Новая функцианальность не добавляется


-По доступу к архитектуре-- BlackBox
-По принципу работы с API-позитивное,негативное
-По запуску кода на исполнение-динамическое
-По степени автоматизации--ручное,частично автоматизаированное
-Примененные техники тест дизайна-классы эквивалентности,граничные значения
-Тестирование является в большей степени исследовательским в виду отсутствия большей части документации
-Уровень тестирования- smoke/critical path/accepted

Критерии успешного выполнения тестирования:
-Smoke Test без 5** и 4** ошибок
-Покрытие каждой функции в виде 1 позитивного и 1 негативного сценария минимум
-При проходе тестсьюита сервис остается работоспособным 
-Отклик на запрос не превышает 1000ms
-Header ответа-согласно требованиям
-Тело ответа на запрос представлен в заявленном виде 
-Наличие всех заявленных параметров для каждой функции
-Схема ответа соответсвует заявленной в требованиях
-Код успешного ответа на запрос согласно заявленному

Тестовая документация:
-Тест план
-Тест Сьюит, Тест кейс, Тест ран
-Баг репорты
-Отчет о результатах тестирования


Риски при проведении тестирования:
-Отсутсвие доступа к API
-Не полное наличие требований и спецификаций тестируемого API

Расписание тестирования:
Smoke_Test(automated)-По готовности
Complete_Test(manual)-По готовности
Acceptеd test(automated)-При защите проекта

Схема тестового окружения:
-Компьютер пользователя(клиент)
-Удаленный сервер(База данных,АPI)
